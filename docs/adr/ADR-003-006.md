## ADR-003 — Python untuk MVP, Go untuk produksi
**Status:** Accepted · 2026-06-30
**Context:** Agen perlu cepat dibangun & mudah di-debug saat MVP, tapi kelak butuh anti-tamper.
**Decision:** v0.1 pakai **Python** (sudah ada di Ubuntu, modular). Refactor ke **Go** (1 binary, sulit ditampering) setelah MVP stabil (TDD §29 v0.3).
**Consequences:** (+) iterasi cepat sekarang; (−) Python lebih mudah dibaca peserta → keamanan berat ditunda ke fase Go (TDD §37).

## ADR-004 — HMAC request signing
**Status:** Accepted · 2026-06-30
**Context:** Agen jalan di mesin peserta → perlu cegah pemalsuan skor tanpa infrastruktur berat.
**Decision:** Setiap request agen ditandatangani **HMAC-SHA256** atas (method+path+body+timestamp+nonce). `agent_secret` **diturunkan** dari `AGENT_HMAC_SECRET` + `participant_id` sehingga server bisa memverifikasi ulang tanpa menyimpan secret per peserta.
**Consequences:** (+) ringan & efektif; nonce/timestamp disiapkan untuk replay-protection penuh (produksi). (−) bukan anti-cheat absolut → dilengkapi eligibility + evidence + verifikasi juara.

## ADR-005 — Vuln pre-baked + baseline per tingkat
**Status:** Accepted · 2026-06-30
**Context:** NAT mencegah injeksi celah dari server; butuh keadilan + bukti before/after.
**Decision:** Semua celah ditanam di image (superset). Server mengaktifkan subset per tingkat. Saat START diambil **baseline snapshot**; check yang sudah lulus saat START → `eligible=false` (anti pre-fix) + dicatat anomali.
**Consequences:** (+) adil, anti curi-start, ada evidence; (−) butuh disiplin menjaga `canonical_baseline_hash` per image (TDD §27).

## ADR-006 — Vercel untuk web
**Status:** Accepted · 2026-06-30
**Context:** Butuh hosting web gratis yang mudah deploy dari GitHub, terpisah dari data.
**Decision:** Web Next.js (UI + API `/v1`) di **Vercel**; data/realtime di Supabase.
**Consequences:** (+) gratis, CI/CD bawaan, cepat; (−) serverless tak bisa server selalu-online → realtime diserahkan ke Supabase.
